# GRD 기반 신규침수 탐지: 방법론과 결과 (2026-07-16)

이 문서는 pre-event baseline 구축부터 7/13~7/14 post-event 전체 프레임을 이용한
신규침수 탐지까지, 이번 세션에서 진행한 작업을 정리한다. 다른 컴퓨터에서
이어서 작업할 수 있도록 스크립트 사용법·설계 근거·결과·한계를 함께 남긴다.

## 0. 요약

- **baseline**: pre-event(6/25~7/3) GRD 5개 날짜, "최신 관측 우선" 합성.
- **post**: 7/13(93FC/3C22/1A5A, 하강)+7/14(FE43 상승, 8EF1/B126 하강) 전체
  6프레임, 픽셀별 "여러 관측 중 가장 어두운 값" 채택으로 탐지를 극대화.
- **분석 범위**: 처음엔 홍수 AOI(충청권)로 제한했으나, "AOI는 다운로드 범위
  선정용이었을 뿐 홍수 모니터링을 그 안으로 제한할 이유가 없다"는 판단에
  따라 **baseline이 커버하는 전체 범위**(약 7.8×6.5도, 남한 대부분+서해+
  북한 일부)로 확장.
- **최종 결과(전체 범위 기준)**: 신규침수(보수적) **243.54 km²**, 신규침수
  (느슨) 325.96 km², 전체 post 수체(baseline 무관, 참고용) 1,999.81 km².
- **⚠️ 중요**: 이 중 **32.6%(79.49 km²)가 북한 지역(위도≥38.3°N)**에서
  나왔다. 북한 산악지대는 레이더 그림자 등으로 오탐 위험이 커서(5절 참고),
  **남한만 보면 164.05 km²**(보수적) / 233.09 km²(느슨)로 좁혀진다. 실제
  홍수 피해 규모를 말할 땐 이 남한 전용 수치를 우선 인용하는 것을 권장한다.

## 1. 배경 문제: baseline과 어떻게 비교할 것인가

RTC 산출물(`downloads/rtc_grd/*_rtc_db.tif`)은 dB 단일 밴드 GeoTIFF이며,
프레임마다 촬영 궤도·시각·범위가 다르다. 홍수 탐지를 위해서는:

1. 홍수 이전 "정상 상태"를 대표하는 **baseline**이 필요하고,
2. post-event 관측을 baseline과 같은 격자에서 비교해 **어디가 새로 물로
   바뀌었는지**를 판정해야 한다.

## 2. Baseline: `build_baseline_composite_grd.py`

이전에는 `gdalbuildvrt`를 터미널에서 직접 호출해 pre-event 합성 VRT를
수동으로 만들었다(재현 불가). 이 스크립트는 그 과정을 `downloads/rtc_grd`의
`*_rtc_db.tif` 원본에서 시작해 전부 자동화한다.

**절차**
1. 파일명의 날짜(YYYYMMDD)를 파싱해 `--cutoff`(기본 `20260703`) 이하만 선택
   → pre-event 5개 날짜: 6/25, 6/26, 7/1, 7/2, 7/3.
2. 날짜별로 프레임을 묶어 모자이크 VRT 생성 (`s1_rtc_db_mosaic_<날짜>.vrt`,
   이미 있으면 재사용).
3. **날짜 오름차순**으로 `gdalbuildvrt`에 넘겨 합성 VRT 생성
   (`s1_rtc_db_composite_latest_pre.vrt`). `gdalbuildvrt`는 나중에 준 파일이
   겹치는 곳을 덮어쓰므로, 오름차순으로 넣으면 **가장 최근 관측이 우선**된다
   (사용자 요청: "중복되면 좀더 최신자료로").
4. 합성 dB에 임계값(`--db`, 기본 -16dB)을 적용해 baseline 수체 지도
   (`baseline_water_latest_grd.tif`) 저장. HAND는 쓰지 않는다(동해안 등 HAND
   미보유 지역 포함, dB만으로 판정).

**결과 (2026-07-15 최초 실행)**: 격자 86,395×72,001px(~10m), baseline 수체
면적 **2,575.22 km²** (5개 날짜 촬영범위 합집합 전체 기준). 7/20에 pre-event
신규 RTC 프레임(6/27·6/28 등)을 반영해 **v2로 재구축**, 7개 날짜 31프레임으로
북한 커버리지 확장.

**v3(진행 중, 7/20): `--fallback-dates` 옵션 추가** — 컷오프(기본 `20260703`)
이하 정식 pre-event 관측이 아예 없는 궤도가 일부 있음을 발견 (예: `1571`
(7/4)↔`9FFF`(7/16), 7/6 계열↔`2B06`(7/18) — 이 궤도들은 6/25~7/3 사이 관측
자체가 없어 그 지역만 baseline이 비어 있었다). `--fallback-dates
20260704,20260706,20260707`로 지정하면 이 날짜들을 `gdalbuildvrt` 인자
**맨 앞(최하위 우선순위)**에 넣어, 정식 기간(≤7/3) 데이터가 있는 곳은 항상
그게 이기고 **빈틈(정식 기간 관측 자체가 없는 지역)만 채우도록** 한다.
"강우가 이미 시작됐을 수 있어 7/4~7/7을 baseline 정식 기간에서 제외"한다는
기존 결정(사용자 지시)과 "북한 전역 커버리지 확보" 요구를 절충한 설계다.

**주의**: pre-event 날짜들이 모내기철과 겹쳐, 논에 물을 댄 상태가 SAR에서
홍수 수체와 유사하게 어둡게 보일 수 있다. baseline에 계절적/농업적 "가짜
수체"가 섞여 있을 가능성이 있다는 점을 감안해야 한다(4절 한계 참고).

**실행**
```bash
conda run -n s1_snappy python build_baseline_composite_grd.py
conda run -n s1_snappy python build_baseline_composite_grd.py --db -18   # 임계값 변경
conda run -n s1_snappy python build_baseline_composite_grd.py \
    --fallback-dates 20260704,20260706,20260707                        # 빈틈 메우기
```

## 3. 신규침수 탐지: `detect_flood_grd.py` → `detect_flood_grd_v2.py`

### 3-A. v1 (`detect_flood_grd.py`) — 7/13 3프레임, AOI 제한, 첫 시도

가장 단순한 형태: baseline과 **7/13 프레임 3개만**(93FC/3C22/1A5A) 홍수
AOI(+0.1도 여백) 격자에서 비교. "보수적" 신규침수 = 아래 3조건 AND:

1. post dB < -16 (절대적으로 수체처럼 어두움)
2. diff ≤ -3dB (baseline 대비 뚜렷한 하락 — speckle이 아닌 실제 변화)
3. baseline dB ≥ -16 (baseline에서 이미 수체였던 곳 제외 — 상시수체/바다
   오판 방지)

**결과**: AOI 내 baseline·post 동시 유효 커버리지 **35.1%**, 신규침수(보수적)
**6.48 km²**. → 사용자 피드백 "결과가 좋지 않네요". 원인 진단: 7/13은 **패스
1개**뿐이라 AOI의 2/3가 애초에 비교 자체가 안 됐다(침수 없음이 아니라
미관측). 최신 촬영을 다 써도 어차피 RTC는 이미 돼 있으니 프레임을 더
합치자는 방향으로 전환.

### 3-B. v2 (`detect_flood_grd_v2.py`) — 7/13+7/14 6프레임, 최대 탐지

**v1과의 차이**
- post 관측을 하나가 아니라 **7/13(하강)+7/14 09:31(상승)+7/14 21:31(하강)
  3개 패스, 6프레임 전체**를 사용.
- 여러 관측이 겹치는 픽셀은 latest-wins가 아니라 **가장 어두운(dB가 가장
  낮은) 값을 채택**한다. 촬영 기하(승/하강)에 따라 같은 수체도 다르게 보일
  수 있는데, 최솟값을 쓰면 "어느 한 패스에서라도 물처럼 보이면 물"로 간주해
  탐지를 극대화한다("수체를 가능하면 많이 잡아보죠" 요청 반영).
- 판정을 3단계로 나눠 함께 출력(느슨한 것부터):
  1. `total_post_water`: post 최솟값 dB < 임계값 (baseline 무관, 참고용)
  2. `new_flood_relaxed`: `total_post_water` AND NOT baseline_water
     (하락폭 조건 없음 — 최대 탐지)
  3. `new_flood_strict`: `new_flood_relaxed` AND diff ≤ -3dB (v1과 동일한
     보수적 조건, 비교용)
- `--dates` 옵션으로 날짜 조합을 자유롭게 지정 가능 (예: `--dates 20260714`
  로 7/14만 분석). 기본 조합(7/13+7/14)은 기존 파일명을 유지하고, 다른
  조합은 날짜를 이어붙인 접미사를 쓴다.

**분석 범위 변경 (핵심 설계 결정)**: 처음엔 v1처럼 홍수 AOI+0.1도 여백으로
제한했는데, QGIS로 확인한 사용자가 "14일 영상의 촬영 범위보다 결과가
현저히 작다"고 지적했다. 확인해보니 AOI 격자(~0.98×1.02도)가 실제 위성
촬영범위(~7.8×6.5도)의 극히 일부였다. 사용자 설명: **"AOI는 영상 다운받을
때 쓰려던 것"**이지 홍수 모니터링 범위를 그걸로 제한할 이유가 없고, "남한
전역 수재해 모니터링을 위해서는 위성영상이 확보되는 모든 지역에 대한 분석이
필요"하다는 방향 제시. 이에 따라:

- AOI 클리핑(`aoi_bbox`/`build_target_grid`) 로직을 완전히 제거.
- 분석 격자를 **baseline 합성(`s1_rtc_db_composite_latest_pre.vrt`) 자체의
  전체 범위**로 대체. baseline이 없는 곳은 어차피 신규침수 판정 기준이 없어
  `valid=False`로 자동 제외되므로, 격자를 넓혀도 로직 변경 없이 안전하게
  "위성영상이 확보되는 모든 지역"을 커버하게 된다.
- 위도 범위가 넓어져(약 33.8~40.2°N) 픽셀당 실면적(km²)이 위도에 따라 달라
  지므로, 전역 단일 `center_lat` 대신 **청크(행 블록)마다 중심위도로 재계산**
  하도록 수정.

| 버전 | post 프레임 | 분석 범위 | 커버리지 | 전체 post 수체 | 신규침수(느슨) | 신규침수(보수적) |

(*) 최종 버전의 24.5%는 AOI 버전의 35.1%/95.7%와 분모가 다르다 (AOI 버전은

늘었다. AOI 밖에서도 상당한 규모의 신규침수가 관측된다는 뜻이다 — 애초에
AOI(충청권)가 홍수 피해 전체를 대표하지 못했을 가능성을 시사한다.

**실행**
```bash
conda run -n s1_snappy python detect_flood_grd_v2.py                    # 7/13+7/14, 전체범위(기본)
conda run -n s1_snappy python detect_flood_grd_v2.py --dates 20260714   # 7/14만
conda run -n s1_snappy python detect_flood_grd_v2.py --db -18 --drop -4 # 임계값 조정
```

## 4. 핫스팟(침수 위치) 분석

`flood_water_strict_20260713_14.tif` / `flood_water_relaxed_20260713_14.tif`
(둘 다 신규 전체범위 버전)에서 2km 격자로 침수 픽셀을 집계해 상위 지역과
GeoJSON 포인트 레이어를 뽑았다 (`flood_hotspots.py`).

**전체 통계 (strict 기준)**: 침수 픽셀 3,069,251개, 뚜렷한 2km 셀 18,660개.
GeoJSON에는 3px 이상인 15,876개 셀을 점으로 저장(`flood_hotspots_strict.geojson`,
relaxed 버전은 `flood_hotspots_relaxed.geojson`, 18,400개 셀).

### 남/북한 분리 집계

상위 핫스팟을 보면 위도 38°N 이상(휴전선 이북)이 다수 섞여 있어, 위도
38.3°N을 기준으로 남/북한을 나눠 재집계했다:

| | 남한(위도<38.3) | 북한(위도≥38.3) | 합계 |
| --- | --- | --- | --- |
| 신규침수(보수적, strict) | **164.05 km²** (67.4%) | 79.49 km² (32.6%) | 243.54 km² |
| 신규침수(느슨, relaxed) | **233.09 km²** (71.5%) | 92.86 km² (28.5%) | 325.96 km² |

북한 비중이 결코 작지 않다(약 1/3). 5절의 이유로 북한 쪽 수치는 신뢰도가
낮으므로, **홍수 피해 규모를 보고할 때는 남한 수치(164.05 / 233.09 km²)를
우선 인용**할 것을 권장한다.

것을 권장한다.

## 5. 한계 및 주의사항

0. **북한 지역 신호는 특히 주의 (가장 중요)**: 4절에서 확인했듯 신규침수의
   32.6%(strict 기준 79.49km²)가 북한 지역이다. 이 수치를 곧이곧대로 "북한
   홍수 79km²"로 해석하면 안 되는 이유:
   - baseline이 되는 pre-event 관측이 북한 지역은 **6/26 단일 패스**에만
     의존하는 경우가 많아(다른 pre-event 날짜들은 남한 위주), 반복관측으로
     검증되지 않은 1회성 값이다.
   - 북한 북부/산악 지역은 지형이 험해 **레이더 그림자(radar shadow)**가
     흔하고, 이 프로젝트의 min-across-observations 설계(3-B절)는 "여러
     관측 중 하나에서라도 어두우면 수체"로 판정하므로, 촬영 기하(입사각)
     차이로 그림자가 지는 위치가 패스마다 달라지면 실제로는 물이 아닌데도
     "신규침수"로 잡히기 쉽다.
   - 이 프로젝트는 애초에 **남한 충청권 홍수(7/8)**를 모니터링하려는
     목적으로 설계됐고, 북한 지역에 대한 별도 검증(현지 보도, 다른 위성
     자료 대조 등)은 하지 않았다.
   - 다만 실제로 북한도 같은 시기 몬순 강우의 영향권일 수 있어, 진짜 홍수가
     섞여 있을 가능성 자체를 배제할 순 없다 — "확인되지 않았다"이지
     "거짓이다"는 아니다. **정확한 결론을 내리려면 북한 지역만 별도로
     반복관측 기반 baseline을 만들거나, 다른 자료로 교차검증이 필요하다.**
1. **모내기철 오탐 가능성**: pre-event 날짜(6/25~7/3)가 모내기 시기와 겹쳐,
   논에 물 댄 상태가 SAR에서 홍수 수체와 유사하게 보일 수 있다. baseline에
   이런 "계절적 수체"가 섞여 있으면, (a) 그 논이 이후 다른 상태(모 성장 등)로
   바뀔 경우 오히려 "감소"로 잘못 나타나거나, (b) 다른 논이 여전히 물을 댄
   상태라 baseline에서 이미 수체 처리돼 정작 홍수가 나도 "신규"로 잡히지
   않을 수 있다.
2. **"전체 post 수체"는 침수량이 아니다**: `flood_water_total_*.tif`
   (1,999.81km²)에는 서해·큰 강·저수지 등 원래 있던 수체가 다 포함된다.
   침수 규모를 말할 때는 반드시 `flood_water_strict`(또는 `relaxed`)를
   써야 한다.
3. **느슨 vs 보수적**: `relaxed`(325.96km²)는 하락폭 조건이 없어 baseline이
   비수체였던 곳이 post에서 한 번이라도 -16dB 밑으로 내려가면 전부 포함한다
   (경계값 근처 speckle 변동에 취약). `strict`(243.54km²)가 더 신뢰할 만한
   하한값에 가깝다.
4. **min-across-observations의 함의**: 여러 패스 중 가장 어두운 값을 쓰므로,
   촬영 기하(입사각)에 따라 원래도 어둡게 보이는 지형(레이더 그림자, 특정
   각도의 평탄한 지표)이 여러 관측 중 하나에서라도 -16dB를 넘으면 수체로
   오판될 수 있다. "최대 탐지"를 우선한 설계이므로 과대추정 방향의 편향이
   있을 수 있음을 감안해야 한다.
5. **처리 중 메모리 제약**: 전체범위 실행은 86,395×72,001px 격자에 6~7개
   래스터를 청크마다 재투영하는 무거운 연산이다. 이 세션에서는 동시에 다른
   SNAP `gpt` 프로세스가 15~18GB를 점유하는 상황에서도 안전하게 돌리기 위해
   `CHUNK_ROWS`를 작게(256~512) 유지했다. 여유 메모리가 충분하면 이 값을
   늘려 처리 속도를 높일 수 있다.
6. **QGIS 파일 잠금**: Windows에서는 QGIS가 레이어를 열어둔 상태로 같은
   파일을 덮어쓰려 하면 "Permission denied"가 난다. 재실행 전 QGIS에서 해당
   레이어를 닫아야 한다.
