# Sentinel-1 SAR 전처리 워크스테이션 RAM 증설 요청 근거

**작성일**: 2026-07-14
**대상 시스템**: Sentinel-1 SAR RTC(Radiometric Terrain Correction) 전처리용 워크스테이션
**작성 목적**: 현재 RAM 용량이 SNAP(ESA SAR 처리 소프트웨어) 배치 처리의 병목임을
실측 데이터로 제시하고 증설을 요청

---

## 1. 현재 시스템 사양

| 항목 | 값 |
| --- | --- |
| CPU | Intel Xeon W-2123 @ 3.60GHz (물리 코어 4개 / 논리 스레드 8개) |
| RAM (설치) | **32GB** (16GB × 2, DDR4-2666, Hynix) |
| RAM 슬롯 | **총 8슬롯 중 2개만 사용 중 — 6슬롯 여유** (기존 모듈 교체 없이 증설 가능) |
| 저장장치 | C: SSD(Samsung 870, 1TB) + D/E/F: HDD(Seagate 4TB × 3) |

## 2. 실측: 씬 1개 처리만으로도 메모리 여유가 바닥남

2026-07-14, Sentinel-1 GRD 씬 **1개**를 RTC 전처리(SNAP `gpt`, 전체 씬·AOI 서브셋
없음)하는 동안 측정:

| 시각 | 여유 RAM | 비고 |
| --- | --- | --- |
| 처리 시작 직후 | 5.2 GB | 총 31.7GB 중 |
| 처리 중(약 20분 후) | **3.7 GB** | 계속 감소 추세 |
| 해당 `gpt` 프로세스 자체 메모리 사용량(Working Set) | **18.49 GB** | 씬 1개만 처리하는데 총 RAM의 58%를 단독 사용 |

- SNAP 처리 그래프에는 타일 캐시 `-c 14G`가 지정돼 있는데, 이 캐시 크기 자체가
  이미 시스템 전체 RAM(32GB)의 거의 절반입니다. 씬 1개만 돌려도 OS·기타 프로그램이
  쓸 여유가 3~5GB 수준으로 줄어듭니다.
- **배치는 씬을 동시에 여러 개 처리하지 못하고 반드시 순차 처리**해야 합니다 —
  씬 1개가 이미 18.5GB를 쓰므로 32GB 시스템에서는 2개를 동시에 돌릴 메모리 여유가
  전혀 없기 때문입니다.

## 3. 실측: 처리 속도

과거 GRD 씬 14개를 순차 처리한 완료 시각 기록을 바탕으로 계산한 씬당 소요 시간:

| 항목 | 값 |
| --- | --- |
| 씬당 평균 처리 시간 | **약 45분** |
| 최소~최대 | 11분 ~ 88분 (씬 크기·DEM 캐시 여부에 따라 편차) |
| 오늘(7/14) 신규 씬 1개 처리 시간 | 약 55분 |

한 번의 홍수 모니터링 캠페인에서 처리해야 하는 씬이 GRD 17개 + SLC 14개(총 31개)
수준이며, 현재 속도로는 **전체 배치 처리에 하루 이상** 소요됩니다.

## 4. 병목 원인 분석

1. **메모리 부족으로 인한 순차 처리 강제** — RAM이 넉넉하면 씬 여러 개를
   동시에(병렬) 돌려 전체 배치 시간을 코어 수(8스레드)만큼 단축할 수 있으나,
   현재는 씬 1개가 메모리를 거의 다 써서 병렬화가 불가능합니다.
2. **타일 캐시 크기 제약** — SNAP의 타일 캐시(`-c` 옵션)를 RAM 부족 때문에 14GB로
   제한하고 있습니다. 캐시가 작으면 같은 데이터를 디스크에서 반복해서 다시 읽어와야
   해 처리 속도가 느려집니다.
3. CPU(8스레드)는 이미 최대로 활용 중이라 **CPU를 늘리는 것보다 RAM을 늘리는 쪽이
   비용 대비 효과가 큽니다.**

## 5. 요청 사항

- **RAM 64GB 이상으로 증설** 요청 (현재 32GB → 64GB, 가능하면 128GB)
- 8슬롯 중 2개만 사용 중이므로 **기존 모듈 교체 없이 추가 장착만으로 증설 가능**
  (동일 규격 DDR4-2666 RDIMM/UDIMM 확인 필요)
- 기대 효과: 씬 병렬 처리 가능(배치 전체 처리 시간 대폭 단축), 타일 캐시 확대로
  씬당 처리 속도 개선, SLC(더 무거운 처리) 배치도 안정적으로 처리 가능
